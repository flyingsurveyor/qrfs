# QRFS Format Notes

This document describes the **current on-wire and on-page format** used by QRFS.
It is written as a practical reference for implementers, auditors, and anyone who wants
to understand what the software actually emits today.

This is **not yet a frozen standard**. It is a descriptive document for the current
implementation in this repository.

## Scope

QRFS turns one input file into:

1. a packed payload (`QFSP`)
2. an optional cryptographic envelope (`QFSC`)
3. one or more QR chunks (`QRC1` / `QRC2` / `QRC3`)
4. printable pages or PNG page images containing Base45 text

The manifest JSON generated by the encoder is **helpful**, but it is **not required**
for decoding or recovery.

## Design goals

QRFS is designed as a physical, offline transport and storage layer.
It can be used for archival backup, but it is also intended for hostile,
intermittent, censored, air-gapped, or highly delayed environments.

Its structure therefore separates:

- file packaging
- protection (clear, password, or public-key)
- chunking and optional FEC
- QR carrier encoding and page layout

## End-to-end pipeline

The encoder currently performs the following steps:

1. Read one input file.
2. Pack it into a `QFSP` payload with metadata.
3. Optionally protect that payload with a `QFSC` cryptographic envelope.
4. Split the resulting blob into QR chunks (`QRC*`).
5. Optionally add FEC parity chunks.
6. Encode each chunk as Base45 text.
7. Place those Base45 strings into QR symbols.
8. Arrange the QR symbols on PDF or PNG pages.

The decoder performs the inverse operations.

## 1. File packaging layer: `QFSP`

The packed file payload begins with the ASCII magic:

```text
QFSP
```

Current version: **1**

### Binary layout

```text
magic         4 bytes   ASCII "QFSP"
version       1 byte    unsigned
metadata_len  4 bytes   big-endian unsigned integer
payload_len   4 bytes   big-endian unsigned integer
metadata      N bytes   UTF-8 JSON
payload       M bytes   raw file bytes or zlib-compressed file bytes
```

The current struct format used by the code is:

```text
> B I I
```

after the 4-byte magic.

### Metadata JSON

The metadata JSON currently contains:

- `filename`
- `mime_type`
- `compressed` (boolean)
- `original_size`

Compression is optional and currently uses **zlib**.
Compression is only kept if the compressed form is actually smaller.

### Notes

- `QFSP` always contains exactly one logical file.
- For directories or multiple files, the current workflow is to archive them first
  into ZIP or another container, then feed that file to QRFS.

## 2. Cryptographic transport layer: `QFSC`

The packed payload may be left clear, encrypted with a password,
or encrypted to a recipient public key.

The cryptographic transport blob begins with the ASCII magic:

```text
QFSC
```

Current version: **5**

### Supported modes

- mode `0`: clear
- mode `1`: password
- mode `2`: public-key

### Shared fields

All `QFSC` blobs begin with:

```text
magic    4 bytes   ASCII "QFSC"
version  1 byte
mode     1 byte
flags    1 byte
```

Current flag bits:

- `0x01` = signed with Ed25519

If signed, the sender metadata is embedded in the header and a 64-byte Ed25519
signature is appended to the end of the blob.

### Signing metadata

When signing is enabled, the header includes:

```text
signer_key_id   8 bytes
verify_key     32 bytes
```

Total signer metadata size: **40 bytes**

The final signature appended at the end is:

```text
signature      64 bytes   Ed25519 signature
```

The signature covers the entire unsigned transport blob.

### Password mode

Password mode uses:

- KDF: **Argon2id**
- Cipher: **AES-256-GCM**
- Header authenticated as AEAD additional authenticated data (AAD)

Layout:

```text
magic      4 bytes
version    1 byte
mode       1 byte   = 1
flags      1 byte
salt       16 bytes
nonce      12 bytes
signer     40 bytes if signed
ciphertext+tag   remaining bytes
signature  64 bytes if signed
```

Current Argon2id parameters in the code:

- output key length: 32 bytes
- opslimit: 3
- memlimit: 64 MiB

Current minimum password length enforced by the code: **14 characters**.

### Public-key mode

Public-key mode uses:

- X25519 recipient key
- libsodium `SealedBox` to wrap a random 32-byte session key
- AES-256-GCM for the payload

Layout:

```text
magic        4 bytes
version      1 byte
mode         1 byte   = 2
flags        1 byte
key_id       8 bytes   recipient key id
sealed_len   2 bytes   big-endian unsigned integer
nonce       12 bytes
signer      40 bytes if signed
sealed      sealed_len bytes   sealed session key
ciphertext+tag   remaining bytes
signature   64 bytes if signed
```

The `key_id` is the first 8 bytes of SHA-256 of the recipient public key.

### Clear mode

Clear mode is not encrypted but still uses the `QFSC` transport container:

```text
magic      4 bytes
version    1 byte
mode       1 byte   = 0
flags      1 byte
signer     40 bytes if signed
payload    remaining bytes
signature  64 bytes if signed
```

### Notes

- Even in clear mode, a sender may still sign the payload.
- AES-GCM authenticates the transport headers via AAD.
- The output of the crypto layer becomes the input to chunking.

## 3. Chunk layer: `QRC1`, `QRC2`, `QRC3`

Each QR code carries exactly one chunk blob.
The chunk blob is then Base45-encoded before being placed into a QR symbol.

### File identifier

Every chunk carries a randomly generated `file_id` of **16 bytes**.
All chunks belonging to the same encoded object must share the same `file_id`.

### Data vs parity

Chunks can be:

- data chunks
- parity chunks

The current implementation uses these integer values:

- `kind = 0` → data
- `kind = 1` → parity

### FEC type codes

The current implementation uses these integer values:

- `0` → no FEC
- `1` → XOR
- `2` → Reed-Solomon

### Legacy versions

QRFS can still parse earlier chunk versions:

- `QRC1` → no explicit FEC metadata
- `QRC2` → XOR-era metadata
- `QRC3` → current format

The current encoder emits **`QRC3`** when FEC metadata is needed.
If no FEC is used and no grouping metadata is needed, older layouts may still be parsed,
but the current implementation path is centered on `QRC3` for modern use.

### `QRC1` layout

```text
magic        4 bytes   ASCII "QRC1"
version      1 byte
file_id     16 bytes
index        4 bytes   global data-chunk index
total        4 bytes   total number of data chunks
payload_len  2 bytes
payload      N bytes
```

### `QRC2` layout

```text
magic        4 bytes   ASCII "QRC2"
version      1 byte
file_id     16 bytes
kind         1 byte
index        4 bytes
total        4 bytes   total number of data chunks
payload_len  2 bytes
group_index  4 bytes
group_size   2 bytes
payload      N bytes
```

`QRC2` is mainly relevant to the XOR grouping era.

### `QRC3` layout (current)

```text
magic         4 bytes   ASCII "QRC3"
version       1 byte
file_id      16 bytes
kind          1 byte    0=data, 1=parity
fec_type      1 byte    0=none, 1=xor, 2=rs
index         4 bytes   global data index for data chunks
total         4 bytes   total number of data chunks
payload_len   2 bytes
group_index   4 bytes   group number
group_size    2 bytes   nominal number of data chunks per group
parity_count  2 bytes   configured parity chunks per group
parity_index  2 bytes   which parity chunk this is inside the group
payload       N bytes
```

Notes:

- `total` always means **data chunk count**, not total QR count.
- For data chunks, `index` is the global position in the logical data stream.
- For parity chunks, `index` is currently set to the `group_index` by the encoder.
  Recovery must therefore use the chunk header fields, not the human footer label.

### Chunk payload size

The QRFS UI exposes a `chunk_size` target.
This is the target payload size used for data chunks before Base45 and QR encoding.

With FEC disabled:

- each data chunk carries up to `chunk_size` bytes of transport blob data

With FEC enabled:

- data chunks still carry up to `chunk_size` bytes
- parity chunks carry their own metadata plus a parity payload

## 4. FEC layer

QRFS currently supports:

- none
- XOR parity
- Reed-Solomon parity

FEC works **per group**.
The current UI only offers group sizes from this set:

- `0` (disabled)
- `2, 3, 4, 5, 6, 8`

### XOR FEC

XOR adds exactly **1 parity chunk** per group.

For a group of `N` data chunks:

- the parity payload stores the original lengths of the group members
- the parity bytes are the XOR of all group payloads padded to `chunk_size`

Parity payload format:

```text
count         2 bytes   number of data chunks actually present in this group
length[0]     2 bytes
length[1]     2 bytes
...
length[n-1]   2 bytes
xor_bytes     chunk_size bytes
```

XOR can recover **one missing data chunk per group**.
If two or more data chunks are missing in the same group, XOR is not enough.

### Reed-Solomon FEC

RS adds `P` parity chunks per group.

Current implementation constraints:

- `P` must be between `1` and `4`
- `P < group_size`

For each byte offset inside a group, the encoder forms a column of `k` symbols
from the padded data chunks, runs Reed-Solomon encoding, and writes the generated
parity symbols into `P` parity vectors.

Each RS parity chunk uses the same length-prefix metadata structure as XOR:

```text
count         2 bytes
length[0]     2 bytes
...
length[n-1]   2 bytes
parity_bytes  chunk_size bytes
```

RS recovery works on **known erasures**.
That means the decoder must know which data or parity chunks are missing.
This is natural in QRFS because chunk indexes and parity indexes are explicit.

A group can recover up to `P` total erasures inside that group.
Those erasures may be:

- missing data chunks
- missing parity chunks
- or a mix of both

Example:

- group size `5`
- parity count `2`

The group carries `5 data + 2 parity`.
It can recover from any combination of up to **2 missing chunks total** within that group.

### Practical note

Parity chunks are slightly larger than plain data chunks because they also carry
per-chunk length metadata. In difficult real-world conditions, parity QR may therefore
be a little denser than data QR.

## 5. Base45 carrier encoding

Before QR symbol generation, each chunk blob is encoded with Base45.

The current alphabet is the RFC 9285 Base45 alphabet:

```text
0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ $%*+-./:
```

Why Base45 is used here:

- it is text-based
- it fits QR alphanumeric mode well
- it works better with generic scanners and browser-side workflows than raw binary payloads
- it remains inspectable and easier to move across software layers that expect text

Base45 is not intrinsically denser than Base64 as a generic encoding.
Its advantage here is that it maps well to the QR symbol model used by QRFS.

---

## 6. QR symbol generation

Each chunk is converted to Base45 text and then encoded as one QR symbol.

The current implementation uses:

- QR version: automatic fit
- border: 2
- internal box size: 3 for generated images before page placement
- selectable ECC level: `L`, `M`, `Q`, `H` in the QR library mapping,
  though the UI currently emphasizes the levels it exposes in presets

QRFS also has its own optional FEC at the chunk layer.
That FEC is independent from the QR symbol's built-in Reed-Solomon ECC.

---

## 7. Page layout

Current page layout in the PDF/PNG generator:

- page size: A4
- grid: **5 columns × 6 rows**
- total QR per page: **30**

The encoder interleaves chunk order inside each page with a small stride-based
reordering function before drawing the page.
This is a page-layout detail intended to spread nearby logical chunks apart.
It is **not** part of the chunk wire format.

The page footer label currently shows a human counter in the form:

```text
{chunk.index + 1}/{chunk.total}
```

This is convenient for quick orientation but should not be treated as authoritative
for parity semantics.
The authoritative recovery data is the chunk header itself.

---

## 8. Technical manifest JSON

The encoder also writes a manifest JSON file.
This is **not required** for decoding, but it is useful for:

- auditing
- reproducibility
- comparing field output with local expectations
- verifying chunk hashes

The manifest currently includes:

- original filename and MIME type
- original, packed, and encrypted sizes
- SHA-256 of original, packed, and encrypted forms
- chunk target size
- QRFS preset name
- file id
- FEC configuration
- per-chunk SHA-256 and lengths
- chosen QR ECC level

The manifest is a convenience artifact, not a prerequisite for recovery.

---

## 9. Recovery invariants

A valid QRFS reconstruction requires at least the following invariants:

- all chunks belong to the same `file_id`
- all chunks agree on `total`
- enough data and/or parity chunks exist to reconstruct all logical data chunks
- Base45 was decoded without mutation
- the resulting `QFSC` blob passes mode-specific verification and decryption
- the resulting `QFSP` payload has the expected magic and version

## 10. Stability notes

The following items should currently be treated as implementation details rather than
permanent public commitments:

- exact page labels
- exact manifest JSON shape
- the current set of UI presets
- whether future QRFS versions will keep Base45 as the only carrier mode

The following items are closer to core format commitments and are good candidates
for long-term stability:

- `QFSP` packaging magic and versioning
- `QFSC` cryptographic envelope structure
- `QRC*` chunk framing and explicit indexing
- grouped FEC semantics
- text-safe carrier encoding suitable for QR transport
